# v5.0 唐宋迁移审计与合同冻结

本文是唐宋迁移的 v5.0 审计记录和后续实现合同。它依据当前源码和轻量扫描结果编写，只冻结迁移边界，不表示本轮已经完成业务源码迁移。

## 1. 当前状态

- 当前分支：`main`，跟踪 `origin/main`。
- 当前仓库已包含唐宋总提示词：`md/prompt/v5.0-唐宋迁移/codex-v5.0-唐宋aiagent历史策略迁移总提示词.md`。
- 外部参考提示词与当前仓库内提示词行数一致，`cmp -s` 结果一致。
- 本轮继续沿用 `AGENTS.md` 和 `md/test/test.md` 的规则：本机只做轻量文本/配置检查；未授权不跑 Xcode build/test、模拟器、Probe、Smoke、Stage Regression、Dynamic Theater Regression、Full 或性能测试。
- 本阶段不改 Swift 业务源码、不改 Xcode project、不改 JSON 剧本数据。

## 2. v5.0 目标

v5.0 的目标不是把 WWIIHexV0 直接改成唐宋游戏，而是完成可交接的迁移合同：

- 明确保留的引擎资产。
- 标出当前二战语义绑定点。
- 冻结后续迁移的 public API 边界。
- 给 v5.1 多势力和通用回合地基留下可执行入口。
- 记录哪些残留允许短期作为 legacy 兼容，哪些不得进入唐宋默认主路径。

## 3. 必须保留的合同

后续版本不得破坏这些核心合同：

- Hex 仍是战术权威：`HexTile.controller` 和 `Division.coord` 决定真实占领、移动、攻击和单位位置。
- Region 仍是战略聚合层：州府/军州/关隘/仓城只能从 hex 状态聚合，不替代 hex。
- `regionToTheater` 仍是初始/基础战区映射，不是运行时推进层。
- `hexToTheater` 仍是运行时动态战区权威。
- `hexToFrontZone` 仍是部署层动态归属权威。
- 玩家、AI、聊天命令和 MockAI 必须收口到 `Command` / `ZoneDirective -> WarCommandExecutor -> RuleEngine`。
- Legacy Agent D 只作回归参考，不得重新成为默认战争 AI 主路径。
- 不恢复 organization；当前战斗核心仍是 strength、retreat、supply、encirclement。

## 4. 当前二战绑定审计

### 4.1 二元阵营

当前阵营合同仍是二元：

- `WWIIHexV0/Core/Faction.swift:3` 定义 `Faction.germany` / `Faction.allies`。
- `WWIIHexV0/Core/Faction.swift:7` 定义 `Faction.opponent`，只能表达二元敌对。
- `WWIIHexV0/Core/Faction.swift:16` 的显示名仍是 `Germany` / `Allies`。

v5.1 必须把多政权合同引入到这一层，短期可保留 `Faction` 作为 legacy 桥，但主路径敌我判断不能继续依赖 `opponent`。

### 4.2 回合与阶段

当前阶段仍绑定二战角色：

- `WWIIHexV0/Core/GamePhase.swift:3` 定义 `.germanAI`、`.alliedPlayer`、`.resolution`。
- `WWIIHexV0/Rules/CommandValidator.swift:138` 只允许 `.germanAI -> .germany`、`.alliedPlayer -> .allies`。
- `WWIIHexV0/Rules/CommandExecutor.swift:173` 的 end turn 仍固定 `germany -> allies -> germany`。
- `WWIIHexV0/App/AppContainer.swift:647` 和 `WWIIHexV0/App/AppContainer.swift:701` 的 AI 触发仍按 Germany/Allies 判断。
- `WWIIHexV0/Turn/TurnManager.swift:382` 的 AI turn 判断仍按二元阵营和二战 phase。

v5.1 应优先引入 `TurnOrderState` 和玩家/AI 控制权判断，使回合推进不再写死 Germany/Allies。

### 4.3 数据加载和默认剧本

当前默认加载仍是阿登：

- `WWIIHexV0/Data/DataLoader.swift:18` 的 `loadInitialGameState()` 默认读 `ardennes_v0_scenario` 和 `ardennes_v02_regions`。
- `WWIIHexV0/Data/DataLoader.swift:70` 的 `loadArdennesDataSet()` 仍是旧数据集入口。
- `WWIIHexV0/Data/DataLoader.swift:153` 的初始 active faction 根据 `.alliedPlayer` / `.germanAI` fallback 到 `.allies` / `.germany`。
- `WWIIHexV0/Data/DataLoader.swift:270` 附近的数据校验仍要求 German / Allied supply source。
- `WWIIHexV0/Data/DataLoader.swift:330` 附近仍检查 Guderian 分配覆盖 German initial units。

v5.2 前不要把阿登数据删除。v5.1 只应建立兼容层，让 v5.2 可以新增 `tangsong_jianlong_960_*` 数据并切默认入口。

### 4.4 玩家可见 UI 文案

当前主界面仍有二战和现代资源文案：

- `WWIIHexV0/UI/HUDView.swift:17` 显示 `Ardennes V0`。
- `WWIIHexV0/UI/HUDView.swift:47` 到 `WWIIHexV0/UI/HUDView.swift:52` 显示 `Manpower`、`Industry`、`Supplies`。
- `WWIIHexV0/UI/EconomyPanelView.swift:11` 显示 `Economy`。
- `WWIIHexV0/UI/EconomyPanelView.swift:38` 到 `WWIIHexV0/UI/EconomyPanelView.swift:46` 显示 `Manpower`、`Industry`、`Supplies`、`Income MP`、`Income IC`。
- `WWIIHexV0/SpriteKit/BoardScene.swift:511` 显示 `Ardennes V0 Board`。
- `WWIIHexV0/UI/RootGameView.swift:98` 的 accessibility label 仍是 `Ardennes V0 hex board`。

v5.5 前可保留源码类型名，但唐宋默认主路径不得继续显示这些文案。

### 4.5 单位、生产和经济

当前单位和生产仍有二战语义：

- `WWIIHexV0/Core/Division.swift:3` 的 `ComponentType` 包含 `tank` / `motorizedInfantry`。
- `WWIIHexV0/Core/EconomyState.swift:87` 的 `ProductionKind` 包含 `panzerDivision` / `motorizedDivision`。
- `WWIIHexV0/Data/unit_templates.json` 仍有 `Panzer Division`、`Motorized Division`、`Infantry Division` 等显示名。
- `WWIIHexV0/Data/DataLoader.swift:507` 的 fallback components 仍按 panzer / motorized / artillery / infantry 映射。

v5.3 必须先做桥接和显示映射，不能一次性删除 legacy case 造成旧数据无法加载。

### 4.6 AI 人物和默认意图

当前 AI 仍有二战人物和目标：

- `WWIIHexV0/Data/general_agents.json` 使用 `Heinz Guderian`。
- `WWIIHexV0/Data/generals.json` 使用 `Heinz Guderian`。
- `WWIIHexV0/Agents/GameAgent.swift` 仍有 Guderian 注释和 fallback。
- `WWIIHexV0/Agents/MockAIClient.swift` 仍以 Bastogne、armor、roads 作为默认启发式。
- `WWIIHexV0/App/AppContainer.swift:725` 构造 `Allied Mock Commander`。
- `WWIIHexV0/App/AppContainer.swift:754` 自动 commander 名称仍是 `German` / `Allied`。

v5.4 应保留 deterministic fallback，但默认显示和 rationale 要迁移为皇帝/枢密/行营/节度使/州府守臣体系。

## 5. v5.1 public API 合同

v5.1 不应直接改完整玩法。建议新增兼容合同：

```text
PowerId
  ASCII id，例如 power_song、power_northern_han、power_southern_tang。

PowerProfile
  displayName、shortName、kind、playerControl、legacyFactionBridge。

TurnOrderState
  power order、activePowerId、round、phase、playerControlledPowers。

PowerRelation
  allied、tributary、neutral、hostile、atWar、submitting、negotiating。

WarRelationRules
  isHostile(a,b)、isAtWar(a,b)、canTarget(a,b)，替代 Faction.opponent 主路径。
```

短期桥接原则：

- `Faction.germany/allies` 可暂时保留，旧阿登数据继续可读。
- 新唐宋数据必须使用 `PowerId` 作为语义主键。
- 旧 `Faction` 只能作为 legacy bridge 或底层兼容字段，不得作为新文档的产品词。
- 新 UI 和新日志读取 display profile，不直接读 `Faction.displayName`。
- 所有 AI directive 继续落到 `ZoneDirective` / `WarCommandExecutor`，不得新增绕过规则系统的执行入口。

## 6. v5.1 实施边界

建议 v5.1 分两步做，不扩大到剧本数据和 UI 重绘：

1. 合同层
   - 新增 `PowerId`、`PowerProfile`、`TurnOrderState`、`PowerRelation`、`WarRelationRules`。
   - 为现有 `germany/allies` 建 legacy profile。
   - 给 `GameState` 增加或派生 turn order / power profile 访问入口。

2. 回合层
   - 让 `CommandValidator.phaseAllowsCommands` 不再硬编码 Germany/Allies。
   - 让 `CommandExecutor.executeEndTurn` 通过 turn order 推进。
   - 让 `AppContainer.shouldRunAI` / `TurnManager.isAITurn` 基于 active power 是否 AI-controlled。

暂不做：

- 不切换默认数据到唐宋剧本。
- 不重命名所有 `Division` 类型。
- 不删除 `Faction.opponent`，但新路径不得继续新增依赖。
- 不做围城、粮道、天命、外交归附。
- 不跑本机重测试。

## 7. v5.2-v5.5 前置约束

- v5.2 新增唐宋 JSON 前，必须先确认 JSON id 全部 ASCII/pinyin，中文只放展示字段。
- v5.2 切默认剧本时，不得删除阿登 legacy 数据；应通过 loader 参数或 scenario registry 管理默认/legacy。
- v5.3 古代军制必须通过现有 `Command` / `RuleEngine` 生效，不得让围城或归附直接改 hex。
- v5.4 AI schema 必须 Codable，解码失败必须 hold/fallback。
- v5.5 UI 迁移时，`HexTile.controller`、`hexToTheater`、`hexToFrontZone` 的显示读取可以变，但权威来源不能变。
