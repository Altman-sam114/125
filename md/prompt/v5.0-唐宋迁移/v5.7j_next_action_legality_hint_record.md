# v5.7j 下一步有限合法性预校验记录

本文记录唐宋 v5.7j 小切片：在 v5.7a / v5.7g 已有 HUD “下一步”提示和高亮数量提示后，把当前 UI 候选的命令合法性口径进一步贴近 `CommandValidator`。

## 1. 目标

- 唐宋场景下，HUD “下一步”优先显示“规则确认可执行”的当前候选行动。
- 对围城、修城、解围、招降、招抚、攻击和行军候选做有限只读预校验。
- 减少 UI 层手写围城/解围/招降条件导致的提示与真实规则分叉。

## 2. 非目标

- 不新增命令。
- 不提交 `Command`。
- 不调用 `RuleEngine.execute`。
- 不写 `GameState`、`GameState.eventLog`、存档、hex / region / theater / front / deploy 或 diplomacy。
- 不改变 `CommandValidator`、`CommandExecutor` 或 `RuleEngine` 语义。
- 不实现通用 dry-run 系统、完整逐命令教程、规则模拟器或发布级布局验收。

## 3. 实现链路

```text
RootGameView.nextActionHint
  -> 读取 AppContainer.selectedValidatedCommandHint
  -> HUDView 显示规则确认可执行项
  -> 不提交命令，不写状态

AppContainer.selectedValidatedCommandHint
  -> 读取当前 selectedActionDivision / UI 候选目标 / movementHighlights / attackHighlights
  -> 构造底层 Command
  -> CommandValidator.validate(command, in: gameState)
  -> 返回只读提示文案
```

`AppContainer` 的围城、修城、解围和招降候选过滤也收口到 `CommandValidator.validate`，避免 UI 继续复制这些规则条件。招抚候选延续既有 `Command.proposeSubmission -> CommandValidator` 过滤。

主线程负责实现、文档同步、diff 整合、main push 和云端 artifact 核对。

## 6. 验证口径

用户已明确要求：“不要在本地测试，都去云端测试”。因此本轮不跑本地 `swiftc -parse`、`git diff --check`、Xcode build/test、模拟器、Probe、Smoke、Stage Regression、Dynamic Theater Regression 或 Full。

本轮提交后只通过 `origin/main` 的 GitHub Actions 云端重验证核对：

```sh
gh run list --workflow "WWIIHexV0 CI Results" --branch main --limit 5
gh run watch <run_id> --exit-status
gh run download <run_id> --dir /private/tmp/wwiihexv0-c-review-<run_id>
```

## 7. 遗留风险

- 当前只是对 HUD 当前 UI 候选做有限预校验，不是完整规则模拟器。

- 未做小屏、Dynamic Type、VoiceOver、截图或真实设备布局验收。
