# v5.6f 战术候选关系感知记录

本文记录唐宋 v5.6f 小切片：把 v5.6e 已同步到 `TurnOrderState.relations` 的 power 级战争关系，前移到 UI、AI 和 `WarCommandExecutor` 的战术候选生成层，减少非敌对对象被高亮、排序或编译成战争命令的噪声。

## 1. 目标

- 玩家点击攻击、攻击高亮、将领目标区推断先读取 `WarRelationRules.canTarget`。
- `WarCommandExecutor` 的敌强评分、敌方 region 候选、可见敌军、围城目标和战术移动/占领候选先读取 `WarRelationRules.canTarget`，且非己方、非敌对 controller 的 hex 不作为战争移动目的地。
- `ZoneCommanderAgent`、`MarshalBattlefieldSummarizer` 和 `MockAICommander` 的敌情估算与 `weightedRegions` 来源先读取 `WarRelationRules.canTarget`。
- 保持最终执行仍由 `CommandValidator`、`CommandExecutor` 和 `RuleEngine` 校验并修改状态。

## 2. 实现口径

```text
TurnOrderState.relations
  -> WarRelationRules.canTarget
  -> AppContainer attack target / highlight / target zone inference
  -> ZoneCommanderAgent / MarshalBattlefieldSummarizer visible enemy regions
  -> WarCommandExecutor enemyStrength / enemyRegions / visibleEnemyDivision / tacticalDestination
  -> Command
  -> RuleEngine
```

本轮只把已经存在的 power 级敌我关系用于候选过滤。若 `TurnOrderState.relations` 缺少关系，`WarRelationRules.canTarget` 仍按 legacy 兼容返回可攻击；若 v5.6e 保守投影仍为 `.atWar`，宋与割据 legacy faction 之间的战术行为保持原有战争态。

## 3. 非目标

- 不实现吴越、南唐、后蜀、北汉等单国 tactical neutral。
- 不把招抚、归附、谈判候选改成 `canTarget`；`Command.proposeSubmission` 仍通过 `CommandValidator` 的外交规则判断。
- 不交割 hex / region controller，不转换、删除或收编部队。
- 不新增国家级 `PowerId`，不拆分 legacy `.allies/.germany` 部队归属。
- 不改变 `CommandValidator.validateAttack`、`validateBesiege`、`validateDemandSurrender` 的最终规则权威。

## 5. 并发审计

本轮按用户要求使用并发子 Agent 加速：

- UI 审计子 Agent 定位 `AppContainer` 点击攻击、攻击高亮、将领目标区推断和 `GeneralCommandPanelView` 展示风险，并明确招抚候选不能替换为 `canTarget`。
- AI / 执行器审计子 Agent 定位 `WarCommandExecutor`、`ZoneCommanderAgent`、`MarshalBattlefieldSummarizer` 和 `MockAICommander` 的敌强、敌区和可见敌军候选残留。

主线程整合后只改战术敌我候选，保留招抚与外交候选的独立规则口径。

## 7. 遗留风险

- `WarRelationRules` 仍读取 legacy power 级关系；v5.6f 只是把该关系前移到候选层，不能表达单个国家与宋停战而其他割据国家继续战争。
- 归附后的纳土、部队转属、治理政策和国家级 tactical ownership 仍需后续小切片。
- `GeneralCommandPanelView` 仍是展示组件；本轮通过 AppContainer 只传入关系合法的 target，未把 `GameState` 注入 view。
