# v5.3 古代军制、粮草、围城城防与生产显示桥记录

本文记录 v5.3 的已完成切片：先把生产、府库和经济规则日志从二战口径迁到唐宋显示桥，再加入唐宋场景专用的古代兵种战斗修正、粮道供给首轮、围城状态首轮、城防耐久和修城命令首轮。它不是 v5.3 完整收口；自动破城、招降、解围命令、围城 overlay、AI 主动围城、漕运专门 UI 和完整资源 schema 仍留后续切片。

## 1. 目标

- 让唐宋默认路径的生产按钮、排产日志、部署日志和经济结算日志使用唐宋术语。
- 让 `ProductionKind` 保留 legacy enum 名，但提供场景感知显示桥。
- 让 `EconomyResources` 保留 `manpower / industry / supplies` 字段，但唐宋显示为丁口、钱帛、粮草。
- 让生产完成后的新单位命名为厢军、禁军、骑军、攻城器械营，而不是 Infantry / Panzer / Motorized / Artillery。
- 不绕过 `Command -> RuleEngine -> EconomyRules` 生产管线。
- 在不改 `ComponentType` Codable schema 的前提下，为唐宋默认场景加入骑军、弓弩守军、攻城器械营、守军的最小战斗差异。
- 在不新增补给状态 schema 的前提下，让唐宋默认场景的粮仓州府、道路、山林和跨河成本影响 `SupplyRules`。
- 在不绕过 `Command -> RuleEngine -> CommandValidator -> CommandExecutor` 的前提下，加入最小 `besiege` 围城命令、`SiegeState` 记录和 Region 面板读法。
- 在 `SiegeRecord` 上加入向后兼容的 `fortification / maxFortification`，让围城压力先损耗城防，城防归零后才触发断粮压制。
- 增加 `Command.repairFortification(defenderId:targetRegionId:)`，让守方军队在被围州府内消耗行动修城。

## 2. 非目标

- 不重命名 `ProductionKind`、`EconomyResources`、`Division` 或 `ComponentType` 的底层 Codable schema。
- 不实现完整 `relieveSiege` 命令。
- 不实现自动破城、招降、围城 overlay、AI 主动围城、漕运专门 UI 或运河独立地形。
- 不引入全地图永久城防表；城防首轮只存在于活跃 `SiegeRecord`。
- 不新增 `imperialGuard`、`cavalry`、`crossbow` 等 Codable `ComponentType` case；唐宋角色先由现有 id 和组件权重推导。
- 不新增新的 `SupplyState` case；唐宋粮道仍落到现有 `supplied / lowSupply / encircled`。
- 不修改 `WWIIHexV0.xcodeproj/project.pbxproj`。
- 不跑本机 Xcode build/test、模拟器、Probe、Smoke、Stage Regression、Dynamic Theater Regression、Full 或性能测试。

规则日志迁移：

- `EconomyRules.queueProduction`：唐宋场景下写“下达军备令：耗丁口/钱帛/粮草”。
- `EconomyRules.resolveFactionTurn`：唐宋场景下写“府库：收入、耗粮、补员、余”。
- `EconomyRules.advanceProduction`：唐宋场景下写“完成整备粮草”或“部署禁军/厢军/骑军/攻城器械营”。
- `EconomyRules.applyStrategicSupplyShortfall`：唐宋场景下写“粮草见底，已补给军队降为缺粮”。
- `RuleEngine` / `AppContainer`：唐宋场景下生产命令显示为“军备(募禁军)”等，不再显示 `QueueProduction(Panzer Division)`。
- `CommandExecutor`：唐宋场景下固守、准退、结束回合日志改为唐宋口径。

古代兵种战斗修正：

```text
Division.tangSongCombatRoles
  id / legacy production kind / 组件权重 -> imperialGuard / prefectureInfantry / cavalry / crossbowGarrison / siegeEngine / garrison

CombatRules.effectiveAttack
  仅 state.isTangSongScenario 为 true 时走唐宋修正
  cavalry: 进攻平原或道路目标 +15%；进攻城池、关隘、山林 -15%
  siegeEngine: 攻城池/关隘/具名城市 +35%；野战 -25%

CombatRules.effectiveDefense
  crossbowGarrison: 守城池/关隘/具名城市 +2 防御
  garrison: 守城池/关隘/具名城市 +1 防御
  siegeEngine: 野战防御 -1
```

这些修正仍只改变 `CombatRules` 的攻防数值。攻击、反击、撤退、消灭仍由 `CommandExecutor` 和 `RuleEngine` 处理；攻击不会直接占领 hex。

粮道供给首轮：

```text
SupplyRules.effectiveSupplySources
  legacy / 非唐宋 -> 只用 MapState.supplySources(for:)
  唐宋 -> 额外把己方控制、passable、supplyValue >= 4 的 region 作为临时粮仓源

SupplyRules.supplyPathCost
  唐宋场景路径上限 = legacy 7 + 2
  道路 -> 1
  city / fortress -> 1
  plain -> 2
  forest / hill -> 3
  mountain -> 4
  跨河仍 +2

SupplyRules / RegionSupplyRules 敌控判断
  不再用 faction.opponent
  改用 WarRelationRules.canTarget 判断 hostile / atWar 关系
```

粮道仍只改变补给判定。缺粮效果继续复用既有 `SupplyState.lowSupply` 对 attack / defense / movement 的惩罚，包围仍复用 `SupplyState.encircled` 和现有 attrition。

围城城防与修城首轮：

```text
Command.besiege(attackerId, targetRegionId)
  -> CommandValidator.validateBesiege
  -> 目标必须是敌对控制的城池 / 关隘 / 高 supplyValue 粮仓州府
  -> 攻击军队必须在目标州府 displayHexes 的 range 距离内
  -> CommandExecutor.executeBesiege
  -> 写入或更新 GameState.siegeState.records
  -> 累积 pressure
  -> 按攻城器械、骑军、要塞地形等因素损耗 fortification
  -> 标记攻击军队 hasActed，并写 siege 日志

Command.repairFortification(defenderId, targetRegionId)
  -> CommandValidator.validateRepairFortification
  -> 守方军队必须属于当前行动方、未行动、位于己控且被围的目标州府内
  -> 目标必须已有 SiegeRecord，且 fortification < maxFortification
  -> CommandExecutor.executeRepairFortification
  -> 按守军/弓弩守军/禁军/补给状态/要塞地形计算 repairGain
  -> 恢复 SiegeRecord.fortification，标记守方军队 hasActed，并写 siege 日志

SiegeState / SiegeRecord
  targetRegionId
  attackerFaction / defenderFaction
  startedTurn / lastUpdatedTurn
  pressure 0...100
  fortification / maxFortification
  besiegingDivisionIds

旧存档兼容：

```text
SiegeRecord.init(from:)
  -> pressure 缺失时按 0
  -> maxFortification 缺失时按默认 10
  -> fortification 缺失时按 maxFortification - pressure / 10 推导并 clamp
```

CommandExecutor.executeEndTurn
  SupplyRules.updateSupplyStates 后执行 applySiegeSupplyPressure
  若围城仍有有效围困军队且目标仍由原守方控制，pressure >= 10 且 fortification == 0 时把目标州府内 supplied 守方军队降为 lowSupply
  若 pressure >= 10 但 fortification > 0，只记录城防尚存，断粮压力暂未突破
  若原守方失去控制或围困军队消失，解除围城记录
```

围城和修城首轮只记录压力、城防和断粮压力，不自动占领 hex，不修改 `RegionNode.controller`，也不推进 `hexToTheater` / `hexToFrontZone`。破城仍必须通过后续合法移动/占领规则落地。

## 4. 当前边界

- `ProductionKind.panzerDivision` 在唐宋场景显示为“募禁军”，但底层仍使用 legacy case，避免一次性破坏存档和旧阿登数据。
- 生产出来的“禁军”“骑军”“攻城器械营”仍暂时复用 `Division.panzer(...)`、`.motorized(...)`、`.artillery(...)` 的组件权重；唐宋战斗角色通过 id 和组件权重推导，不改变存档 schema。
- 缺粮当前仍沿用 `SupplyState.lowSupply`；粮仓源来自高 `supplyValue` 受控州府，不代表完整漕运、仓储容量或粮草运输队。
- 围城当前有 `besiege`、`repairFortification` 和 `SiegeState` 首轮：没有 `relieveSiege`、自动破城、招降、围城圈 SpriteKit overlay 或 AI 主动围城指令。
- 唐宋战斗修正目前只覆盖最小兵种差异；没有新增水师或战斗 UI 说明。
- EventLog 分类标题仍是 Combat / Supply / Reinforce 等英文；这是 v5.5 UI 视觉和文本收口范围。
