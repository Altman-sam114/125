# v5.8aq 唐宋整补粮道门槛记录

## 1. 本轮目标

在 v5.8ao 写入古代/中古作战方向、v5.8ap 落地骑军追击后，本轮继续强化“粮草补给是唐宋作战核心变量”：

- 唐宋场景下，缺粮或被围军队不能直接通过整补恢复。
- 整补必须要求军队处于 `SupplyState.supplied`，即先接通粮道再恢复战力。
- 拒绝原因显示为“粮道不通，不可整补”。
- 不扩大到完整粮队、漕运、仓储容量、低补给 attrition 或 CK3 式战争系统。

## 2. 并发子 Agent 结论

本轮按用户要求使用并发子 Agent 加速扫描候选方向：

- 子 Agent 建议低补给 attrition 是后续可做的粮草切片，但会改变回合损耗节奏，风险较高。
- 子 Agent 建议围城破防后守军转为 `encircled`，但会加强围城/attrition 连锁，适合作为后续独立切片。
- 子 Agent 扫到 UI / 文档中仍有部分“前线 / 防区”残留，适合后续 v5.8ar 玩家态口径清理。
- 主线程选择本轮最小安全落点：只改唐宋 `Command.resupply` 的校验门槛和显示桥。

## 3. 已完成实现

- `WWIIHexV0/Rules/CommandValidator.swift`
  - `validateRecoveryCommand` 在已有阶段、军队存在、归属和已行动校验之后，新增 `canRecover` 判断。
  - 唐宋场景下仅允许 `division.supplyState == .supplied` 的军队整补。

本轮严格遵守用户要求：不在本地测试，不运行本地 build、Swift parse、Markdown 检查、`git diff --check`、模拟器、截图或 VoiceOver 实机验收。

## 6. 非目标与风险

- 不实现完整 CK3 式战争系统。
- 不新增低补给 attrition、粮队、漕运、仓储容量、会战窗口、骑军 AI 或自动破城。
- 不把围城破防守军自动改为 `encircled`。
- 不重命名 `FrontLine` / `FrontZone` / `hexToFrontZone` 等内部 legacy schema。
- 该规则会让唐宋缺粮军队更难恢复，后续需结合云端构建和实战手感评估是否需要配套粮道修复、补给来源或 AI 整补策略提示。
